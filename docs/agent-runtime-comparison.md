## 1. Executive Summary

**The one core architectural difference is the location of the agentic loop relative to Dapr, and it is a structural *inverse*.**

`dapr-agent-py` subclasses the dapr-agents `DurableAgent` framework (`class OpenShellDurableAgent(DurableAgent)`, `services/dapr-agent-py/src/main.py:1668`) and **decomposes** the agentic loop into Dapr workflow activities ŌĆö each LLM turn is its own `call_llm` activity (`main.py:1844-1853`) and each tool call its own `run_tool` activity (`main.py:1950-1954`), so every step is independently checkpointed, retried (max 8 attempts), hookable, compactable, and observable. `claude-agent-py` instead delegates the **entire** LLM+tool loop to the Claude Agent SDK's `query()` iterator, draining it to completion inside **one** opaque `run_claude_sdk_turn_activity` per turn (`services/claude-agent-py/src/claude_sdk_runner.py:596`, `:679-684`); Dapr provides only a durable session shell plus external-event injection.

Every other divergence ŌĆö durability granularity, retry counts, the four dropped Claude-Code-port subsystems, silently-ignored `agentConfig.mcpServers`, batch-vs-incremental event publishing, and the SWE-bench patch-extraction branch ŌĆö is a **downstream consequence** of that one inversion. This trade-off (fine-grained control/observability/feature-port vs. SDK-native simplicity) is the central tension any standardization must *preserve*, not erase.

## 3. Why These Decisions Were Made (Observed Rationale)

**Why dapr-agent-py decomposes the loop into per-activity steps.** The dapr-agents `DurableAgent` framework drives the loop turn-by-turn; orchestrator bodies replay deterministically from the top, so side-effecting work (MCP stdio/http connect, dict mutation) is *banned* from the body and pushed into activities: *"dict mutations and MCP stdio/http connections are banned from orchestrator bodies (they re-run deterministically on every replay); activities ARE allowed to have side effects"* (`main.py:4736-4740`). The 9 provider adapters are re-applied on every `call_llm` *"to survive durable workflow replay"* (`main.py:2716`). Retry was widened from the framework default ~24s/3 to ~140s/8 *"to cover pod restart + image pull + Dapr sidecar handshake when a worker dies mid-activity"* (`main.py:5892-5896`).

**Why dapr-agent-py monkeypatches `DaprChatClient` instead of using the Dapr Conversation API.** *"This bypasses the Dapr conversation API which has a langchaingo bug where tool_choice is sent as a string instead of a dict"* (`anthropic_adapter.py:4-6`); CLAUDE.md adds the broader reason ŌĆö the Conversation API is still ALPHA. The gateway adapter *"Replaces ~3,622 lines of near-duplicate per-provider HTTP plumbing with one route-by-name shim"* (`gateway_adapter.py:5-6`).

**Why the hook/compaction subsystems live inside the activities (and were not ported to claude-agent-py).** Hooks exist *"to inject behavior into the agent's tool-use loop via the same JSON protocolŌĆ” used by the TypeScript CLI"* (`docs/hooks-and-plugins.md`) ŌĆö and they fire INSIDE `run_tool`, a per-tool durable boundary that simply does not exist in a single-activity SDK loop. Compaction *"runs inline inside the existing call_llm activityŌĆ” No new Dapr activity is registeredŌĆ” This keeps workflow history growth identical"* (`compaction/engine.py:1-17`), mutating durable `entry.messages` (`engine.py:666`). These are intrinsically coupled to the per-activity loop (verdict C9), which is the architectural reason they have no attach point in claude-agent-py.

**Why claude-agent-py is a thin SDK shell.** *"Keep Dapr Workflow as the workflow-builder contract while delegating the coding loop to Claude Agent SDK / Claude Code presets"* (`services/claude-agent-py/pyproject.toml:7-8`). Dapr is the durable shell + orchestrator contract; the loop is the SDK's. The SDK + bundled CLI provide hooks/permissions/autocompact/plugins natively, so claude-agent-py inherits CLI autocompact *for free* ŌĆö but invisibly, because it never wires the SDK callbacks back into the Dapr/BFF layer.

**Why the uniform `session_workflow` bridge dispatch exists.** *"WorkflowŌåöSession bridge is now a structural invariant: every durable/run against any durable agent runtime routes through session_workflow so the run appears in /sessions/{id}ŌĆ” Skipping the bridge for browser-use left the per-agent Deployment unwokenŌĆ” stalling Dapr's CreateWorkflowInstance with context deadline exceeded"* (`sw_workflow.py:1762-1778`). The app-id is deliberately the only routing variable: *"dispatch session_workflow to the selected Dapr app id, which may be a dedicated agent runtime, a shared runtime pool, or a legacy shared app id"* (`sw_workflow.py:1925-1928`).

**Why the deployment control plane was retired.** stacks `9ddaaa4d6` / wfb `788bae9b`: *"Replaces our custom AgentRuntime CRD + Kopf controller with upstream kubernetes-sigs/agent-sandbox primitives"* ŌĆö Arc 1 per-session Sandbox via Kueue, Arc 2 browser warm pools, Arc 3 controller decommissioned. The browser carve-out is performance-driven: *"browser-use-agent and Playwright-MCP agents need a long-lived warm pool (chromium boot Ōēł 3-5 s of every run)"* (`agent-workflow-host.ts:26-29`).

**Why SWE-bench runs the turn inline.** *"Do not add an extra per-turn child agent_workflow for SWE-benchŌĆ” startup-time workflow actor routing/placement churnŌĆ” The current SWE-bench path runs the one-shot turn inline inside the already started session_workflow"* (`docs/swebench-dapr-workflow-operations.md:191-201`); and *"keep the durable direction a stable app-id worker pool"* (`:210-227`) ŌĆö which is why the seeded claude-agent-py canary rides `agent-runtime-pool-coding` (`seed-swebench-fixtures.ts:226`).

## 5. Candidate Seams for a Standardized Runtime Contract

A standardization should draw the abstraction line at the seams that ALREADY carry both runtimes, formalize them, and add explicit *capability declarations* so the platform knows when a swap is lossy rather than silently dropping features.

### 5.1 OrchestratorŌåÆruntime dispatch contract ŌĆö **ALREADY EXISTS (formalize)**
Both runtimes register a Dapr workflow named `session_workflow`, accept the same BFF `childInput`, honor `autoTerminateAfterEndTurn`, and return `{success, output, content, sessionId, agentRuntime, ŌĆ”}`. At the seam the difference reduces to one `app_id` string (verdict C5; `sw_workflow.py:985-1018`, `:1922`). **Recommendation:** replace the string-keyed ladder + scattered if-chains with ONE runtime-descriptor registry (`name ŌåÆ {appId, image, workflowName, capabilities}`) and a versioned `DurableSessionRuntime` contract (named workflow, documented input schema, required return keys, auto-terminate semantics). This is the cleanest abstraction line.

### 5.2 Session-event contract ŌĆö **ALREADY EXISTS (extract + tier)**
The ONE fully-shared seam (verdict C10): identical ingest endpoint, `_CMA_EVENT_TYPE_MAP`, `_SESSION_SUPPRESSED_TYPES`, producer triple, fire-and-forget daemon thread. **Recommendation:** extract the publisher into ONE shared library (today it is near-verbatim duplicated; dapr-agent-py's adds trace stamping + `agent.llm_usage` + Notification hooks the lean claude copy lacks). Standardize a MINIMUM vocabulary (lifecycle `status_*` + `agent.message`/`tool_use`/`tool_result`) PLUS an OPTIONAL incremental tier (`message_delta`, `llm_usage`, `context_usage`, `hook.decision`) so the contract can express that claude-agent-py is batch-only.

### 5.3 AgentRuntime CR / deployment model ŌĆö **ALREADY EXISTS (data-drive it)**
Non-browser runtimes are dispatched identically as per-session Kueue Sandbox pods differing ONLY by image (verdict C12; `agent-workflow-host.ts:313-330`). The custom AgentRuntime CRD + Kopf controller were retired for upstream agent-sandbox + Kueue. **Recommendation:** keep this model ŌĆö it is correct. Add a declarative per-runtime descriptor (image + daprd contract: app-port 8002, `enable-workflow`, Configuration name, registers `session_workflow`) so a new runtime is added by DATA, not by editing the `agentImage` if-chain + container-name allowlists. The browser SandboxWarmPool carve-out (Arc 2) is the only structural non-fungibility and stays separate.

### 5.4 Tool / MCP surface contract ŌĆö **MUST BE BUILT (hardest seam)**
The deepest divergence (verdict C7). **Recommendation:** define a runtime-capability declaration (`supportsMcp`, `supportsBuiltinOpenShellTools`, `supportsHooks`, `supportsPermissionGating`, `ownsSandbox`) so the platform KNOWS when a swap is lossy and can warn/reject rather than silently drop tools. To CLOSE the gap on claude-agent-py without re-decomposing the loop: the Claude SDK natively supports `ClaudeAgentOptions.hooks` + `can_use_tool` + `plugins` + `mcp_servers` + `include_hook_events`. Wiring `agentConfig.mcpServers` into the SDK options and forwarding SDK `HookEventMessage` to `publish_session_event` would restore MCP + hook parity ŌĆö none of that exists today (verdict C8 open question).

### 5.5 Config resolution ŌĆö **PARTIALLY EXISTS (lift the resolver)**
`agentConfig` is already shared on the wire (modelSpec, mcpServers, hooks, plugins, permissionMode, maxTurns, contextStrategy); dapr-agent-py reads all of it, claude-agent-py reads a subset. **Recommendation:** lift model/provider resolution into a shared runtime-agnostic resolver returning `{provider, model, capabilities}` and let each runtime adapt it (Dapr Component name vs SDK model string). This decouples the model catalog from the `DaprChatClient` path and is prerequisite to running any provider on either runtime. Then standardize which `agentConfig` fields are MANDATORY vs runtime-optional.

## 7. The Durability Trade-off Must Inform the Standardization, Not Be Erased by It

This is the crux. `dapr-agent-py`'s per-activity durability gives **strictly stronger guarantees** than `claude-agent-py`'s single-activity SDK loop:

**Where dapr-agent-py is stronger:**
- **Fine-grained replay (verdict C2).** A mid-turn worker death replays completed `call_llm`/`run_tool` activities from durable history; only the failing activity re-runs (up to 8 attempts). On claude-agent-py the same crash re-runs the *entire* SDK `query()` from the prompt (up to 3 attempts) with **no durable record** of intermediate tool results (`claude_sdk_runner.py:604-613`). For a long multi-tool turn this is a real fault-granularity regression.
- **Per-step observability + governance (verdicts C8, C9).** Because each tool call is its own activity, dapr-agent-py can fire PreToolUse permission gating, emit a `hook.decision` event per tool, run inline compaction mutating durable state, and emit the full `claude_code.*` OTEL surface ŌĆö all impossible to attach in a single opaque activity. Swapping to claude-agent-py silently drops every one of these.
- **Incremental durable event stream (verdict C11).** Mid-turn progress is observable and replay-safe on dapr-agent-py; on claude-agent-py it is invisible until the turn completes and batch-publishes.

**Where claude-agent-py is simpler / better:**
- **SDK-native features for free.** The Claude Agent SDK + bundled CLI provide hooks, permissions, plugins, skills, and autocompact internally; claude-agent-py inherits autocompact and the latest SDK behavior with ~1.6k LOC vs dapr-agent-py's ~31k, no provider-adapter maintenance, no `dapr-agents==1.0.3` version-pin coupling.
- **It KEEPS the durable turn-timer** dapr-agent-py removed (verdict C3) ŌĆö a small but real reliability win.

**Implication for standardization.** The contract must **express** this trade-off as a runtime capability, not paper over it. A swap from dapr-agent-py to claude-agent-py is NOT semantically transparent: it changes crash recovery (per-activity ŌåÆ whole-turn), drops MCP tools, drops hook governance, drops the OTEL surface, and coarsens the event stream. The right design:

1. Make the dispatch, event, and deployment seams uniform (┬¦5.1ŌĆō5.3) so any conforming runtime *can* be dispatched.
2. Attach an explicit **capability/guarantee descriptor** to each runtime (`durabilityGranularity: per-activity | per-turn`, `supportsMcp`, `supportsHooks`, `supportsPermissionGating`, `incrementalEvents`, `ownsSandbox`).
3. Let the platform **warn or reject** a swap when the target runtime's declared guarantees are weaker than the agent's requirements ŌĆö instead of today's silent feature drops.
4. Optionally **close specific gaps** on the SDK side (wire `ClaudeAgentOptions.hooks` + `can_use_tool` + `mcp_servers` + `include_hook_events`) where parity is achievable without re-decomposing the loop.

In short: standardize the *seams* (they already mostly exist), but treat per-activity durability vs SDK-single-activity simplicity as a **declared, first-class runtime property** that callers choose with eyes open ŌĆö not a difference the abstraction hides.
